# release — 날짜식 태그로 릴리즈 시작

canalyst-controller 의 릴리즈는 **태그 푸시 한 번**으로 굴러간다. 이 스킬은
사람이 직접 하기 번거롭고 실수하기 쉬운 두 가지를 대신한다.

1. 직전 릴리즈 이후의 커밋을 모아 **릴리즈 노트**(마크다운)를 작성한다.
2. 그 노트를 메시지로 담은 **annotated 태그** `vYYYY.MM.DD` 를 만들어 푸시한다.

태그가 origin 에 올라가면 `.github/workflows/release.yml` 이 트리거되어 코어
(PyInstaller) + UI(electron-builder NSIS·portable) 를 빌드하고, **태그 메시지를
릴리즈 본문 그대로** 써서 GitHub Release 를 만들고 설치본을 첨부한다. 즉 여기서
쓰는 태그 메시지가 곧 릴리즈 노트다 — 깔끔한 마크다운으로 쓸 것.

> **전제**: 태그 푸시 트리거는 *태그가 가리키는 커밋에 존재하는* 워크플로우
> 파일을 실행한다. 따라서 `release.yml` 이 아직 `main` 에 머지되지 않았다면 태그를
> 찍어도 **아무 일도 일어나지 않는다**. 첫 릴리즈 전에 `git ls-tree origin/main
> .github/workflows/release.yml` 로 워크플로우가 main 에 있는지 확인하고, 없으면
> 먼저 PR 로 머지하도록 사용자에게 안내한다.

## 왜 이런 구조인가 (배경 — 어기면 릴리즈가 깨진다)

- **main 은 ruleset 으로 보호**되어 직접 push 가 막혀 있다. 그래서 버전 번호를
  파일에 커밋하는 대신, **태그 버전을 빌드 시점에 주입**한다(워크플로우가
  `electron-builder -c.extraMetadata.version` 으로 설치본 버전을 태그에 맞춤).
  → 이 스킬은 파일을 고치지 않고 **태그만** 만든다. package.json 버전은 안 건드린다.
- **버전 형식은 날짜식 `vYYYY.MM.DD`** (예: `v2026.06.23`). 월/일은 **2자리 0
  패딩**한다(`06`, `09`). 워크플로우가 트리거 패턴을 4-2-2 자리로 매칭하므로
  패딩을 빠뜨리면 트리거되지 않는다.
- **프리릴리즈**는 뒤에 `-rc.N` 을 붙인다(예: `v2026.06.23-rc.1`). 하이픈이 있으면
  워크플로우가 GitHub 에 프리릴리즈로 표시한다.
- 날짜의 선행 0(`06`)은 semver 규칙상 무효라, 워크플로우가 내부적으로
  `2026.6.23` 로 변환해 electron-builder 에 넘긴다. **그래서 설치본 파일명은
  `...2026.6.23.exe`** 가 된다(릴리즈 제목 `v2026.06.23` 과 0 패딩만 다름 — 정상).

## 절차

### 1. 사전 점검

릴리즈는 보통 최신 `main` 의 HEAD 에서 찍는다. 다음을 확인한다.

```bash
git rev-parse --abbrev-ref HEAD     # main 인가?
git status --porcelain              # 비어 있어야(작업트리 깨끗) 함
git fetch origin --tags --prune
git log --oneline -1 origin/main    # 로컬 HEAD 가 origin/main 과 같은가?
```

`main` 이 아니거나, 커밋되지 않은 변경이 있거나, origin/main 보다 뒤처져 있으면
**그 사실을 사용자에게 알리고 어떻게 진행할지 확인**한다(다른 브랜치/커밋에서
릴리즈하려는 의도일 수도 있으니 단정하지 말 것).

### 2. 버전 결정

- 기본값: **오늘 날짜**로 `vYYYY.MM.DD`. (오늘 날짜는 세션 컨텍스트의
  `Today's date` 를 쓴다. 예: 2026-06-23 → `v2026.06.23`)
- 사용자가 특정 날짜·프리릴리즈를 지정하면 그에 맞춘다(`v2026.06.23-rc.1`).
- **충돌 확인**: 같은 태그가 이미 있으면(`git rev-parse <tag>` 성공) 그날 이미
  릴리즈된 것이다. 사용자에게 알리고 → 프리릴리즈 `-rc.N` 로 갈지, 다른 날짜로
  갈지 정한다. 기존 태그를 말없이 덮어쓰지 말 것(`-f` 금지 — 발행된 릴리즈가 꼬인다).

월/일 0 패딩을 반드시 지킬 것: 6월 9일 → `v2026.06.09`.

### 3. 릴리즈 노트 작성 (AI 가 직접 정리 — 이 스킬의 핵심)

이 단계가 스킬의 본체다. **`git log` 를 기계적으로 그대로 옮기는 게 아니다.**
변경들을 *이해해서* 사용자가 읽을 만한 릴리즈 노트로 **편집·요약**하는 것이
사람이 직접 하기 귀찮아 이 스킬에 맡기는 핵심 가치다. 커밋 메시지를 1:1 로
복사하면 이 단계를 한 의미가 없다.

먼저 원재료를 모은다. 비교 기준은 **직전 *정식* 릴리즈 태그**로 잡는다 —
프리릴리즈(rc) 는 `--exclude='*-*'` 로 제외해야, rc 다음 정식 릴리즈의 노트가
"rc 이후 변경 없음"으로 텅 비지 않는다.

```bash
# 직전 '정식' 태그 (하이픈 포함 prerelease 는 제외). 첫 릴리즈면 빈 값 → 전체 이력
LAST=$(git describe --tags --abbrev=0 --exclude='*-*' 2>/dev/null)
if [ -n "$LAST" ]; then RANGE="$LAST..HEAD"; else RANGE="HEAD"; fi
git log $RANGE --no-merges --pretty=format:'%h %s%n%b'    # 제목+본문 모두
```

커밋 제목만으로 무슨 변경인지 모호하면 `git show <hash>` 나, gh 가 있으면
`gh pr view <번호>` 로 맥락을 더 확인한다. 첫 릴리즈는 이력이 길 수 있으니 사소한
것은 묶고 사용자가 체감하는 변화 위주로 추린다.

그다음 **직접 판단해서** 노트를 쓴다:

- **사용자 관점으로 옮긴다.** 커밋 말투("stdin 감시 스레드 인터프리터 크래시
  제거")가 아니라 사용자가 느끼는 효과("앱 종료가 더 안정적으로")로. 내부
  리팩터링·CI 처럼 체감이 없는 변경은 과감히 줄이거나 맨 아래로 내린다.
- **묶고 요약한다.** 관련된 여러 커밋은 한 항목으로 합친다. 1 커밋 = 1 줄이 아니다.
- **의미 단위로 섹션을 나누고 내용에 어울리는 이모지를 붙인다.** 아래는 출발점일
  뿐 — 변경 성격에 맞는 다른 이모지를 써도 좋고, 빈 섹션은 뺀다.
  - ✨ 새 기능 · 🐛 버그 수정 · ♻️ 개선/리팩터링 · ⚡ 성능 · 🎨 UI/디자인 ·
    📝 문서 · 🔧 내부/빌드
- **PR 번호(`#7`)는 항목 끝에 남긴다** — GitHub 가 자동으로 링크해 준다.
- **사실을 지어내지 않는다.** 커밋에 없는 기능·수정을 적지 않는다.

좋은 예 — 커밋을 그대로 옮긴 게 아니라 이해하고 정리·요약·합친 결과다:

```markdown
## ✨ 새 기능
- 창 크기·위치를 기억해 재실행 시 그대로 복원합니다 (#5)
- 송신 프레임 프리셋을 저장하고 덮어쓸 수 있습니다 (#2, #7)

## 🐛 버그 수정
- 앱 종료 시 코어 프로세스와 장비 연결을 확실히 정리해 종료가 깔끔해졌습니다 (#4, #6)
- 고부하 수신 중에도 모니터가 최신 프레임을 계속 따라갑니다

## 🎨 UI
- 폰트를 Pretendard 로 통일하고 다크 테마를 다듬었습니다
```

태그 메시지의 **첫 줄**은 릴리즈를 한 줄로 요약하는 제목으로 한다(예:
`v2026.06.23 — 프리셋 UI 개선 및 종료 안정화`). 그 줄도 릴리즈 본문 맨 위에
그대로 노출되므로 정보가 담기게 쓴다. 첫 줄 다음은 빈 줄, 그 아래 위 섹션들.

### 4. 사용자 확인 (필수)

태그 푸시는 **실제 릴리즈 빌드·발행을 즉시 트리거**하는 외부 영향 작업이다.
되돌리기 번거로우니, 푸시 전에 반드시:

- 만들려는 **태그명**과 **릴리즈 노트 전문**을 보여 주고,
- 어떤 커밋(HEAD) 에 붙는지, 정식/프리릴리즈 여부를 명시하고,
- 사용자의 승인을 받는다.

승인 전에는 태그를 만들지 않는다(로컬 생성도 보류).

### 5. annotated 태그 생성 + 푸시

노트를 임시 파일로 써서 `-F` 로 태그 메시지에 넣으면 줄바꿈·마크다운이 그대로
보존된다(`-m` 여러 개로 짜맞추지 말 것).

```bash
# 노트를 파일로 저장 (예: $TMP/release_notes.md)
git tag -a "v2026.06.23" -F "$TMP/release_notes.md"
git push origin "v2026.06.23"
```

태그는 브랜치가 아니므로 보호 ruleset(main push 차단)과 무관하게 푸시된다. 만약
태그 푸시 자체가 거부되면 태그 보호 ruleset 이 있는 것이니 사용자에게 알린다.

### 6. 마무리 안내

푸시 후:

- `release` 워크플로우가 시작됐음을 알리고, 진행 상황을 보여 준다.
  `gh run watch` 또는 `gh run list --workflow=release.yml -L 1` 로 추적할 수 있다.
- 빌드가 끝나면 `gh release view <tag> --web` 으로 릴리즈 페이지를 연다.
- 빌드는 코어 PyInstaller + Electron 패키징까지 도므로 수 분 걸린다 — 기다리는
  동안 사용자가 다른 일을 할 수 있게 안내한다.

## 빠른 예시

**입력:** "오늘자로 릴리즈하자"
**동작:** 오늘=2026-06-23 → 태그 `v2026.06.23` 후보. 직전 태그 이후 커밋으로 노트
작성 → 태그명·노트 제시 → 승인 후 `git tag -a … -F …` + `git push origin v2026.06.23`
→ 워크플로우 추적 안내.

**입력:** "릴리즈 후보 하나 올려줘, rc로"
**동작:** `v2026.06.23-rc.1` (그날 rc 가 이미 있으면 `-rc.2`). 노트 작성·확인·푸시.
워크플로우가 GitHub 에 **프리릴리즈**로 발행.
